# 公共部署的存储、事件与 Secret 边界

## 一、适用范围

本文冻结 VulcanModelRouter 从本地单机模式扩展到公共、多实例部署时必须替换的边界。它不绑定某一家云厂商，也不允许通过猜测配置自动选择云服务。

本地默认模式保持以下组合：

- SQLite 保存业务元数据、执行、事件、目录和租约。
- Windows DPAPI 保护用户级 Secret 文件。
- 本地私有目录保存不可变资源对象。
- 持久事件轮询负责 SSE 等待与重放。

SQLite 和本地对象目录不构成多实例共享部署。公共部署必须显式注入满足下列契约的实现；未注入时不得宣称具备多实例能力。

## 二、Secret 信封加密

### 2.1 组合方式

公共部署必须按以下顺序组合：

1. 部署层实现 `internal/secret.KeyWrapper`，连接操作员明确选择的 KMS、HSM 或受管密钥库。
2. 使用 `secret.NewEnvelopeProtector(keyWrapper)` 创建信封保护器。
3. 使用 `secret.NewProtectedStore(secretDirectory, protector)` 创建持久 Secret Store。
4. 业务数据库只保存 `secret://local/...` 不透明引用，不保存明文、访问令牌、刷新令牌或上游任务私有句柄。

每个 Secret 使用独立随机 AES-256 数据密钥和 Nonce；数据密钥只以外部密钥系统返回的密文形式进入信封。信封头中的精确密钥版本、包装密文与 Nonce 一并通过 AES-GCM 附加数据认证。

### 2.2 KeyWrapper 必须满足的行为

- `WrapDataKey` 只接受 32 字节随机数据密钥，并返回不可变、可长期定位的精确密钥版本 ID 与非空包装密文。
- `UnwrapDataKey` 必须使用记录中的精确密钥版本，返回恰好 32 字节；不得用“当前密钥”尝试解密历史信封。
- 密钥轮换必须保留历史版本的解密能力，直到所有旧信封完成重包或过期删除。
- 实现必须设置请求超时、取消传播与有界重试；不得把明文数据密钥、包装密文、Secret 内容或云错误响应正文写入日志。
- Router 进程身份只授予所需包装/解包权限；不得授予创建、删除或导出主密钥权限。
- 外部密钥系统不可用时，Secret 读写必须关闭失败；禁止回退到明文文件、环境变量主密钥或进程内固定密钥。

`internal/secret/envelope_test.go` 验证唯一数据密钥、完整往返、元数据认证、篡改拒绝和输入大小边界。具体部署实现还必须用其真实 SDK 增加 KeyWrapper 集成测试。

## 三、共享数据库契约

公共多实例数据库实现必须同时满足 `execution.Store` 与 `execution.LeaseStore`：

- 幂等键按调用面所有者隔离，重复请求只返回原记录，不重新解析可变 Target。
- Record CAS 更新与有序事件追加必须在同一事务提交；任一事件非法时不得修改记录或事件序列。
- 事件序号必须连续且只增，`ListEvents` 必须按序稳定返回。
- `ListRecoverable` 只返回非终态执行，并保持稳定创建顺序。
- 执行租约必须支持排他获取、当前所有者续约、精确所有者释放和到期接管。
- 数据库错误与 Context 取消必须原样形成显式失败，不得把未知提交结果当成成功重试。

`internal/sqlitestore/execution_store_conformance_test.go` 是共享实现的准入矩阵。新增 PostgreSQL、CockroachDB 或其他共享数据库之前，必须让同一套 Store 与 Lease 断言运行在新实现上；不得只复制 SQLite 的单元测试。

## 四、跨实例事件分发

持久数据库中的事件表是唯一事实源。`execution.PollingEventDistributor` 提供不依赖进程内通知的基线：任意实例都可以根据所有者、执行 ID 与最后序号读取新增事件并续播 SSE。

部署可用数据库通知或消息总线替换等待机制，但通知只能作为“可能有新事件”的唤醒信号：

- 消息不得直接成为公开事件事实。
- 收到通知后仍须从 `execution.Store.ListEvents` 读取并校验有序持久事件。
- 通知丢失、重复或乱序不得影响重放正确性。
- 客户端断开不取消执行；重连继续使用最后事件序号。

## 五、资源对象存储

资源元数据与对象字节必须分离。公共部署实现 `resource.ObjectStore`，保持以下语义：

- `Publish` 从已完成暂存对象原子产生一个不可变对象键；已存在键返回冲突，不覆盖。
- `Open` 只按 Router 生成的不透明对象键读取，不接受外部 URL 或任意路径。
- `Delete` 幂等；元数据状态转换仍由 Resource Metadata Store 的 CAS 负责。
- 对象存储凭据、桶内部地址、预签名 URL 与物理对象键不得进入公开 VCP 资源结构。
- 上传、读取与删除必须传播 Context，并配置大小、超时、MIME 与租户配额限制。

事件、数据库、Secret 和对象存储必须由同一租户/项目边界装配；不得用共享对象目录绕过元数据层的所有权校验。

## 六、上线准入

公共多实例部署上线前必须同时满足：

1. 新共享数据库通过完整 Store/Lease 一致性矩阵与竞态测试。
2. 真实 KeyWrapper 通过信封集成测试、密钥轮换演练和 KMS/HSM 故障演练。
3. 对象存储通过不可变发布、并发冲突、取消、删除幂等与租户隔离测试。
4. 任意实例重连 SSE 时，只依赖持久事件即可完成重放与后续等待。
5. 执行所有者进程崩溃后，只有一个新实例在租约到期后接管。
6. 日志、指标、Trace、错误和审计记录均通过敏感信息扫描。

上述任一项没有部署实现或验证证据时，系统只能以单机模式运行，不能通过配置文字宣称已具备公共多实例能力。
